# Policies

## Стиль ответа
- Коротко и по делу.
- Сначала факты и риски, затем решение.
- Для утвержденного плана: Plan Fidelity, реализация, затем Verification status.
- Не гадать там, где репозиторий позволяет проверить факт.

## Обязательный аудит перед implementation-изменениями
1. Определить тип scope: upstream code, modular code, maps, tgui, build/CI, docs.
2. Проверить текущее состояние: файлы, include graph, зависимости, существующие hooks, тесты, совместимость.
3. Явно зафиксировать, что не меняется.
4. Для нетривиальной задачи без готового пользовательского плана подготовить decision-complete план без "решим по ходу".
5. Сформулировать критерии приемки и команды проверки.
6. Если пользователь дал или утвердил план, сначала оформить/подтвердить active task-state contract и выполнить plan-mapping challenge по `WORKFLOW_RULES.md`.

## Challenge-block для нетривиальных задач
Нетривиальные задачи: рефакторинг, перенос логики, изменение контрактов, архитектурные изменения, сложные модульные интеграции.

Для таких задач без готового пользовательского плана обязательно включать:
1. Минимум 3 содержательных сомнения.
2. Проверку допущений: что подтверждено фактами, что остается гипотезой.
3. Минимум 2 альтернативы реализации.
4. Обоснование выбранной альтернативы и причины отказа от остальных.
5. Список рисков и конкретный способ закрытия каждого риска.

No-Go:
- Если сомнения не закрыты фактами из репозитория или тестируемыми гипотезами, реализация не начинается.
- Если план уже утвержден пользователем, незакрытое сомнение, которое меняет `MUST/KEEP/REJECT`, переводит challenge в `BLOCKED`; нельзя заменять план альтернативной реализацией без явного согласия пользователя.

## Принципы реализации
- Modular first: новая бизнес-логика по умолчанию идет в `modular/**`; явно утвержденный пользователем путь не менять молча, отклонение фиксировать в `DECISIONS.md` или согласовывать.
- Upstream diff должен быть минимальным и локальным, но это не разрешает оставлять старое ядро production-reachable/callable вопреки утвержденному rewrite/remove/replace плану.
- Предпочтение отдается modular injectors, hooks, adapters и другим extension points; переписывание существующих upstream-proc рассматривается только как крайняя мера.
- Если upstream все же приходится переписывать, diff должен оставаться минимальным, причина такого решения должна быть явно понятна из контекста задачи, а разметка выполняется по правилам `SS220 EDIT`.
- Старые строки сохраняются в комментариях только там, где это требует `SS220 EDIT` policy.
- Внутри `modular/**` не добавляются `SS220`-комментарии о модульности.
- Не редактировать вручную generated/compiled artifacts, если это не прямой scope задачи.
- Любой структурный перенос делать через совместимые интеграционные точки, если это возможно и не подменяет утвержденный план wrapper/fallback решением.

## Анти-паттерны
- "Решим по ходу" без decision-complete плана.
- Выполнение малого hotfix/wrapper/fallback вместо утвержденного rewrite/remove/replace плана.
- Закрытие `MUST/KEEP/REJECT` только тестом или compile-pass без diff-level evidence.
- Инвазивная правка апстрима без проверки modular-варианта.
- Несвязанные рефакторинги в апстрим-файлах.
- Игнорирование build/CI контрактов, по которым живет репозиторий.
- Финализация задачи без проверки диффа и без явного статуса по верификации.

## Definition of Done
Задача завершена, если:
1. Нет незакрытых решений внутри scope.
2. Изменения локальны и читаемы.
3. Документация обновлена там, где поменялись правила или контракты.
4. Для утвержденного плана все `MUST/KEEP/REJECT` имеют diff-level evidence и не имеют статуса `PARTIAL`, `BLOCKED` или `DEVIATED`.
5. Выполнены релевантные проверки или явно указан Verification status и почему проверки не запускались.
6. Ответ содержит фактический список изменений, Plan Fidelity/Verification status и остаточные риски, если они есть.
